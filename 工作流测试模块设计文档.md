# 工作流测试模块 - 前端设计文档

## 设计理念

参考 n8n 的用户体验设计，核心原则：**简单直接，减少干扰**。

---

## 交互流程

### 方式1: 测试整个工作流
```
1. 用户点击顶部"运行测试"按钮
   ↓
2. 前端直接调用 /api/workflow-tests/run (不传 testInput)
   ↓
3. 显示测试结果面板，开始轮询状态
   ↓
4. 画布上各节点显示执行状态
```

### 方式2: 从指定节点测试
```
1. 用户悬停在某个节点上
   ↓
2. 节点右上角出现绿色小播放按钮
   ↓
3. 用户点击该按钮
   ↓
4. 前端调用 /api/workflow-tests/run (传入 startNodeId)
   ↓
5. 从该节点开始执行
```

### 方式3: 右键菜单（备选）
```
1. 用户右键点击节点
   ↓
2. 选择"从该节点测试"
   ↓
3. 执行方式同方式2
```

## UI 组件说明

### 1. 画布节点状态徽章

位置：节点右上角

| 状态 | 颜色 | 图标 | 说明 |
|------|------|------|------|
| PENDING | 灰色 | ⏳ | 等待执行 |
| RUNNING | 蓝色 | 🔄 | 执行中（旋转动画） |
| COMPLETED | 绿色 | ✅ | 执行成功 |
| FAILED | 红色 | ❌ | 执行失败 |
| CANCELLED | 黄色 | ⛔ | 已取消 |

### 2. 节点测试按钮（悬停显示）

- 位置：节点右上角
- 外观：绿色圆形播放按钮
- 显示时机：节点未在测试状态中且已保存草稿
- 点击：直接触发从该节点测试

### 3. 测试结果面板（右侧滑出）

内容：
- 整体执行状态徽章
- 执行摘要：总耗时、成功节点数、失败节点数
- 节点执行列表（按执行顺序）
- 选中节点详情：输入数据、输出数据、配置快照、错误信息

## 数据流转

### 测试输入

**当前设计：不要求用户输入**

- `testInput` 字段不传或传 `null`
- 后端使用节点配置作为执行依据
- 将来可以扩展：支持保存默认输入、从结果面板修改后重测

### 测试执行状态轮询

使用**指数退避策略**：
- 初始间隔：500ms
- 每次 ×1.5
- 最大间隔：2000ms
- 停止条件：状态为 `COMPLETED` / `FAILED` / `CANCELLED`

## 设计思路讨论点

### 1. 测试输入数据来源

**当前前端方案：**
- 不强制要求用户输入
- 直接使用节点配置执行

**可扩展方案（将来）：**
- 在测试结果面板添加"修改输入重测"按钮
- 弹出对话框让用户编辑
- 保存为该工作流的默认测试输入

### 2. 节点执行顺序

前端按 `executionOrder` 字段排序显示节点列表。

### 3. startNodeId 的语义

需要确认：
- 是"从该节点开始执行"（包含该节点）？
- 还是"从该节点的后一个节点开始"？

**前端期望：从该节点开始执行（包含该节点）**

### 4. 测试数据持久化

问题：测试执行记录是否需要持久化？
- 前端可以在刷新后调用 `/api/workflow-tests/draft/{draftId}/latest` 恢复
- 建议保留最近一次的测试结果

## 前端已实现的接口调用

```typescript
// 运行测试
workflowApi.runWorkflowTest({ draftId, startNodeId? })

// 获取执行详情
workflowApi.getTestExecution(testExecutionId)

// 获取草稿的最近一次执行
workflowApi.getLatestTestExecution(draftId)

// 清理资源
workflowApi.cleanupTestResources(draftId)
```

## 总结

当前设计以**简单直接**为优先，去掉用户输入步骤，让用户一键就能测试。后续可以根据需求逐步添加高级功能。
